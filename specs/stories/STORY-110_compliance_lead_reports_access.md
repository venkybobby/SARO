# STORY-110: Grant Compliance Lead Access to Reports (P1 Mapping Fix)
Status: ready
Screen/Area: RBAC / Persona→Tab Mapping (`Sidebar.jsx:17`, reports routes)

## Goal
The primary buyer persona — the one who must hand evidence to a regulator — can reach TRACE View and AIMS evidence packs but cannot reach the report generator; only risk_officer and admin can. Add `reports` to the compliance_lead mapping with full backend authorization parity.

GRC mapping: EU AI Act record-keeping/reporting duties of deployers; persona value-path integrity for the primary buyer.

## Acceptance Criteria (Given/When/Then)
- AC-1: Given role `compliance_lead`, When the sidebar renders, Then the Reports tab is visible.
- AC-2: Given role `compliance_lead`, When they generate any report type available to risk_officer, Then the backend authorizes the request (nav and API parity).
- AC-3: Given a report generated by compliance_lead, When the audit trail is inspected, Then the generation event records actor, persona, report type, and parameters.
- AC-4: Given the Persona→Tab Mapping Matrix, When updated, Then the doc reflects the change and the RBAC parity test passes.

## Edge Cases
- Report types with operator-only data sources → either include with read-only data or exclude with an explicit in-UI note; no silent partial reports.
- Secondary issue noted in review (Upload & Scan on compliance persona blurring the operator boundary) → log as a finding for production-posture review; do not change in this story.

## Out of Scope
- New report types; report template changes.

## Non-Functional Requirements
- Standard project rules.
